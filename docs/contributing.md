# Правила участия

## Формат коммитов

Для каждого коммита используйте Conventional Commits, включая коммиты, которые
делает агент:

```text
<type>[optional scope]: <description>
```

Примеры:

```text
feat(admin): добавить редактор доступности
fix(public): перезагружать слоты после конфликта бронирования
test(e2e): покрыть гостевой сценарий бронирования
ci: запускать интеграционные тесты в GitHub Actions
docs: описать процесс релизов
```

## Разрешенные типы

| Тип | Когда использовать |
| --- | --- |
| `feat` | Пользовательское поведение продукта или новые возможности API. |
| `fix` | Исправления ошибок. |
| `test` | Автоматические тесты и тестовые вспомогательные функции. |
| `ci` | GitHub Actions и автоматизация релизов. |
| `docs` | Только документация. |
| `chore` | Обслуживание проекта без изменения поведения. |
| `refactor` | Внутреннее изменение структуры кода без изменения поведения. |
| `build` | Система сборки, зависимости или метаданные пакетов. |

Ломающие изменения должны использовать `!` после типа или области либо footer
`BREAKING CHANGE:`:

```text
feat(api)!: переименовать поля payload бронирования
```

## Коммиты агента

Коммиты агента подчиняются тем же правилам, что и коммиты человека. Не
используйте `agent:` как тип коммита. Выбирайте тип, который описывает изменение:

- `test(e2e): добавить публичный сценарий бронирования`
- `ci: добавить release-please workflow`
- `docs: описать сценарии проверки`

Если изменение смешивает несвязанные типы работ, по возможности разделите его на
несколько коммитов до merge. Заголовок squash-коммита тоже должен быть
Conventional Commit, потому что release-please читает коммиты, попавшие в
`main`.
