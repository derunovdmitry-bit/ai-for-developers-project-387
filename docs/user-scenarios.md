# Пользовательские сценарии для проверки

Эти сценарии задают минимальный набор ручных и автоматических проверок для MVP
бронирования слотов в календаре. Backend остается источником истины для
бизнес-правил; frontend-проверки покрывают реальный пользовательский путь и
базовую UX-валидацию.

## Матрица сценариев

| ID | Сценарий | Ожидаемый результат | Автоматическое покрытие |
| --- | --- | --- | --- |
| US-01 | Гость открывает публичную страницу бронирования при пустом календаре. | Профиль владельца виден; если типов встреч нет, страница показывает состояние пустого списка вместо элементов бронирования. | Сборка frontend и backend-тесты публичного API. |
| US-02 | Владелец входит в админку с настроенным паролем. | Backend устанавливает `admin_session`; владелец попадает в защищенную панель управления. Неверный пароль возвращает ошибку. | `tests/e2e/booking-flow.spec.ts`, `backend/tests/auth.test.ts`. |
| US-03 | Владелец создает типы встреч и конкретные окна доступности. | Созданные типы встреч становятся публичными; валидные будущие окна генерируют слоты. Невалидные интервалы и неавторизованные запросы отклоняются backend. | Backend-тесты типов встреч и доступности; подготовка E2E-данных использует тот же API. |
| US-04 | Гость выбирает тип встречи, свободный слот и отправляет имя, email и необязательный комментарий. | Backend создает бронирование со статусом `confirmed`, вычисляет `endsAt`, а frontend показывает подтверждение. | `tests/e2e/booking-flow.spec.ts`, `backend/tests/bookings.test.ts`. |
| US-05 | Гость отправляет слот, который стал недоступен или находится вне 14-дневного окна бронирования. | Backend отклоняет запрос доменной ошибкой; frontend очищает устаревший слот и может перезагрузить список слотов. | `backend/tests/bookings.test.ts`. |

| US-07 | Владелец пытается удалить или сузить данные, у которых есть будущие подтвержденные бронирования. | Backend блокирует удаление типа встречи и изменения доступности, которые сделали бы будущие бронирования невалидными. | `backend/tests/event-types.test.ts`, `backend/tests/availability.test.ts`. |
| US-08 | TypeSpec-контракт генерируется заново. | Генерация OpenAPI проходит успешно; при необходимости mock API можно запустить из сгенерированной схемы. | CI запускает `npm run openapi:generate`; Prism доступен через `npm run mock:api`. |

## Область интеграционного тестирования

В репозитории используются два уровня автоматических интеграционных проверок:

- Backend-интеграционные тесты используют Fastify `inject()` и покрывают
  доменные правила без открытия сетевых портов.
- Playwright-интеграционные тесты запускают реальные backend и frontend, а затем
  проверяют вход в админку и путь гостевого бронирования в Chromium.

Playwright-тесты намеренно создают тестовые данные через admin HTTP endpoints
перед проверкой гостевого UI. Так тесты остаются сфокусированными на полном пути
бронирования и не становятся хрупкими из-за внутренних деталей date-picker.
